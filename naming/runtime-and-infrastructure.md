# Runtime and Infrastructure Resources

Runtime names appear in deployment commands, routing, secret grants and
recovery procedures. Favor stable service purpose over implementation detail.

## Infrastructure Projects

An infrastructure project, account workspace or state boundary SHOULD follow:

```text
<domain>-<purpose>[-<environment>]

platform-network-prod
platform-observability
portfolio-edge-stg
```

Include `prod` or `stg` when the project is an isolation, access-control,
billing or state boundary. Omit it from shared tooling with explicit
multi-environment scope.

```text
Good                              Avoid
platform-network-prod             terraform-prod-final
platform-observability            misc-infra
```

## Docker Swarm Stacks

A stack is the environment-scoped application deployment unit:

```text
<system>-<environment>

orders-prod
orders-stg
observability-prod
```

Docker Swarm prefixes deployed service, network and volume names with the
stack name. Therefore, use concise logical service names inside the stack:

```yaml
services:
  api:
  worker:
  db:
```

Resulting operator-visible service names include `orders-prod_api` and
`orders-prod_worker`. The underscore is generated by Swarm; it is not the
preferred delimiter for manually named runtime resources.

| Good | Avoid | Reason |
| --- | --- | --- |
| Stack `orders-prod`, service `api` | Stack `prod`, service `orders-api-prod` | Avoid duplication and retain ownership |
| Stack `observability-prod` | Stack `obs1` | Readable during incidents |

## Services

Outside a namespace or stack that already supplies system scope, use:

```text
<system>-<component>

orders-api
orders-worker
portfolio-web
```

Inside a scoped stack or Kubernetes application boundary, the short component
name MAY be used if tooling and dashboards preserve the parent identity.

Service names MUST remain stable through releases. Use image tags and telemetry
attributes for version, and labels for environment or instance detail.

## Networks

Name networks by trust boundary or traffic purpose, not implementation:

```text
<scope>-<purpose>[-<environment>]

edge-public-prod
orders-private-prod
observability-monitoring-prod
```

| Good | Avoid | Reason |
| --- | --- | --- |
| `edge-public-prod` | `traefik-net-new` | Purpose survives proxy replacement |
| `orders-data-prod` | `network2` | Reveals permitted traffic class |

An externally shared ingress network should carry a platform scope such as
`edge`, not the name of a single consuming application.

## Secrets

Secret object identifiers MUST be safe to show in configuration, logs and
access reviews; secret values never belong in names.

```text
<system>-<consumer>-<credential>[-<environment>]

orders-api-db-password-prod
orders-api-jwt-signing-key-prod
edge-proxy-dns-token-prod
```

Include the environment if the secret store or cluster may contain more than
one environment. Map secret objects into concise runtime variables:

```text
Secret object: orders-api-db-password-prod
Application key: DATABASE_PASSWORD
```

Avoid names that expose account identifiers, actual hosts, employee names or
rotation history:

```text
Avoid
password-for-carlos
db-password-new-final
aws-key-AKIA...
```

Rotation replaces secret content or a controlled version reference; it should
not cause unbounded naming drift.

## Domains and Subdomains

DNS names use lowercase labels separated by `.`. Use subdomain hierarchy to
express endpoint and environment:

```text
Production:      api.example.com
Staging:         api.stg.example.com
Preview:         pr-184.preview.example.com
Operations:      grafana.ops.example.com
```

Do not place `prod` in the public production URL unless multiple production
realms genuinely require disambiguation. Users and integrations should see the
canonical endpoint.

```text
Good                              Avoid
api.stg.example.com               api_staging.example.com
grafana.ops.example.com           new-grafana-prod-final.example.com
```

## Hosts and Cluster Nodes

Physical or virtual nodes need stable infrastructure identity, independent of
the workloads scheduled on them:

```text
<location>-<platform>-<role>-<ordinal>

br-sp1-swarm-manager-01
br-sp1-swarm-worker-03
us-east1-runner-linux-02
```

Use a documented region or site code and zero-padded ordinals where multiple
nodes exist. Do not name nodes after the current application, IP address or an
individual. A node may host different services over its lifetime.

When represented as Kubernetes `Node` objects, these names remain compatible
with DNS-subdomain naming constraints.
